서비스 소개

신혼부부의 장기 재무 설계를 자동화하는 AI 모바일 금융 서비스입니다. 일반적으로 재무 설계는 전문가 상담이 필요한 복잡하고 진입장벽이 높은 영역이지만, BeHappy는 챗봇과의 자연스러운 대화 형태로 이를 풀어냈습니다.

사용자가 챗봇 온보딩을 통해 소득·자산·부채·소비 패턴 등을 입력하면, AI가 결혼·출산·내집마련·은퇴 등 생애주기 이벤트에 맞춰 맞춤형 재무 로드맵과 액션 플랜을 생성합니다. 실시간 스트리밍 응답, 음성 입력, 강력한 데이터 검증, 모바일 퍼스트 UX를 통해 복잡한 금융 도메인을 누구나 사용할 수 있는 인터페이스로 구현하는 것이 핵심 목표였습니다.

SSE & Whisper 기반 실시간 스트리밍 인프라

문제 정의

AI가 재무 분석 리포트를 한 번에 생성하는 방식은 응답까지 5~10초의 대기 시간이 발생하여 사용자가 빈 화면을 응시하며 이탈하는 문제가 있었습니다. 특히 모바일 환경에서는 키보드 입력으로 수십 개의 재무 정보를 입력해야 했기에 입력 피로도가 높아 온보딩 완료율이 저조했습니다. 재무 상담이라는 신뢰가 중요한 도메인에서 "AI가 멈춘 것 같은" 인상은 치명적이었습니다.

기술적 판단 및 해결

OpenAI Chat Completions API의 stream 옵션을 활용해 토큰 단위로 응답을 전송하고, SSE(Server-Sent Events) 프로토콜로 클라이언트에 실시간 푸시했습니다. EventSource API의 한계(POST 요청 미지원)를 우회하기 위해 fetch + ReadableStream을 사용한 커스텀 SSE 클라이언트를 구현했고, 네트워크 끊김 시 자동 재연결과 부분 응답 누적 처리를 핸들링했습니다. 입력 측면에서는 OpenAI Whisper API를 모바일 마이크와 연동해 음성→텍스트 변환 파이프라인을 구축했고, 60대 이상 사용자도 키보드 없이 정보를 입력할 수 있게 했습니다.

결과 및 성과

첫 토큰 응답까지의 시간(TTFT)을 500ms 이하로 단축했고, 사용자는 AI가 "실시간으로 생각하는" 듯한 체험을 통해 평균 8~10초의 응답 대기를 자연스럽게 받아들이게 됐습니다. 음성 입력 옵션은 모바일 입력 부담을 크게 줄여, 디지털 친숙도가 낮은 사용자층의 온보딩 완료율 개선에 기여했습니다.

느낀점

스트리밍 응답에서 진짜 중요한 건 데이터를 단순히 받는 게 아니라 토큰 손실, 부분 JSON 파싱, 재연결 등 엣지 케이스를 견고하게 다루는 것임을 체감했습니다. UX 측면에서는 "응답이 끝나는 속도"보다 "응답이 시작되는 속도"가 사용자 체감에 훨씬 큰 영향을 준다는 점이 인상 깊었고, AI 서비스에서 스트리밍은 단순한 기술 선택이 아니라 핵심 UX 자산이라는 인식을 갖게 됐습니다.

기여도: 프론트엔드 100% (SSE 클라이언트, Whisper 통합, 에러 처리 모두 단독 구현)

Zod 기반 동적 스키마 검증 시스템

문제 정의

AI가 응답하는 JSON에는 재무 분석 결과·로드맵·추천 액션 등 310개 이상의 동적 필드가 포함됐습니다. AI 응답은 본질적으로 비결정적이라 필드 누락, 타입 불일치(숫자 문자열, null 등), 예상치 못한 값이 빈번하게 발생했고, 이를 그대로 렌더링하면 화이트스크린이나 잘못된 정보 노출로 이어지는 치명적 문제였습니다. 기존의 단순 typeof 검사로는 중첩된 필드 간 의존성(예: "대출 있음" 체크 시 대출 금액 필수)을 검증할 수 없었습니다.

기술적 판단 및 해결

Zod를 도입해 모든 AI 응답과 사용자 입력에 스키마 기반 검증을 적용했습니다. 특히 superRefine과 discriminatedUnion을 활용해 크로스 필드 검증(예: 대출 보유자의 잔액·이율 필수, 자녀 계획자의 출산 시기 필수 등)을 선언적으로 표현했습니다. 실패 시 사용자 친화적 에러 메시지를 자동 생성하는 ErrorMap을 작성했고, AI 응답 검증 실패는 별도 로깅 채널로 모아 프롬프트 개선에 활용했습니다.

결과 및 성과

런타임 에러가 크게 감소했고, AI 통신 계층에서 발생하던 알 수 없는 크래시가 거의 사라졌습니다. 개발자는 검증된 데이터를 신뢰하고 다음 로직을 작성할 수 있어 디버깅 시간이 단축됐고, AI 응답 형식이 변경되더라도 스키마 한 곳만 수정해 전체 시스템 안정성을 유지할 수 있었습니다.

느낀점

AI처럼 본질적으로 예측 불가능한 외부 응답을 다룰 때 런타임 검증은 단순한 옵션이 아니라 필수 인프라임을 절감했습니다. 또한 검증 로직을 코드 곳곳에 흩뿌리지 않고 스키마라는 단일 진실 공급원(Single Source of Truth)으로 모으면 유지보수성이 극적으로 개선됨을 경험했습니다.

기여도: 프론트엔드 100% (Zod 스키마 설계 및 통합 단독 구현)

데이터 기반 제품 피벗 (GA4 & Mixpanel)

문제 정의

서비스 출시 초기 타겟 사용자층과 사용 패턴에 대한 정량적 근거가 부족해 다음 제품 방향성을 정하기 어려웠습니다. 팀 내부의 가설(20~30대, PC 위주)과 실제 사용자 행동 간 괴리가 있을 가능성이 높았고, 이를 검증할 데이터 인프라가 미비한 상태였습니다.

기술적 판단 및 해결

GA4에 사용자 세그먼트별·플로우별 이벤트를 설계해 온보딩 단계별 이탈 지점, 디바이스, 연령대를 추적했고, Mixpanel로는 챗봇 응답 만족도와 사용자 행동 퍼널을 분석했습니다. 직접 이벤트 네이밍 컨벤션을 만들고 데이터 레이어를 구축해 분석 일관성을 확보했고, 분석 결과를 매주 팀에 공유하며 가설→실험→검증 사이클을 만들었습니다.

결과 및 성과

60대 이상 사용자가 예상보다 큰 비중을 차지하고 있다는 사실과, 전체 접속의 95%가 모바일에서 발생한다는 점을 발견했습니다. 이는 기존 가설을 완전히 뒤집는 발견이었고, 이를 근거로 모바일 퍼스트 UX 전략으로 제품 방향을 피벗했습니다. 폰트 크기, 터치 영역, 음성 입력 등 시니어 친화적 요소를 우선순위화하는 의사결정으로 이어졌습니다.

느낀점

"느낌"이 아니라 데이터로 의사결정하는 경험을 통해, 개발자가 단순히 코드를 작성하는 사람이 아니라 제품 방향성에 영향을 줄 수 있는 핵심 역할임을 체감했습니다. 특히 분석 인프라를 처음부터 잘 설계해두면 이후 모든 의사결정의 질이 달라진다는 점이 큰 배움이었습니다.

기여도: 분석 설계·전략 제안 주도

40+ 커스텀 훅 & Recoil 중앙 상태 관리

문제 정의

금융 데이터, 챗봇 플로우, UI 상태(모달·로딩·에러)가 컴포넌트 곳곳에 흩어져 있어 동일한 상태를 여러 곳에서 동기화하는 코드가 반복되고 있었습니다. 특히 챗봇 단계와 사용자 입력 데이터가 강하게 결합되어 있어, 한 부분 수정 시 다른 부분이 깨지는 의존성 문제가 잦았고, 컴포넌트가 자체적으로 비즈니스 로직을 들고 있어 테스트와 재사용이 어려웠습니다.

기술적 판단 및 해결

Recoil의 atom과 selector를 활용해 surveyState(170+ 필드)와 selfState(140+ 필드)로 도메인을 분리하고, 총 310개 이상의 필드를 중앙 관리했습니다. 상태 변경은 stateHandlerSelector(Redux-like 액션 패턴)와 selfSpouseSelector(본인/배우자 데이터 병합) 같은 selector를 통해 일관되게 처리했습니다. 컴포넌트에서 직접 atom을 구독하지 않고, 도메인별로 추상화된 40개 이상의 커스텀 훅(useChatFlow, useFinanceData, useUserProfile 등)으로 비즈니스 로직과 UI를 분리해, 훅 내부에서 파생 상태와 사이드 이펙트를 캡슐화했습니다.

결과 및 성과

같은 데이터를 사용하는 컴포넌트가 추가될 때마다 새로운 보일러플레이트 없이 훅 한 줄로 끝낼 수 있어 개발 속도가 크게 향상됐습니다. 비즈니스 로직 변경 시 훅만 수정하면 모든 사용처에 일관되게 반영돼 버그가 줄었고, 신규 기능 개발 시 기존 훅을 조합하는 방식으로 확장이 쉬워졌습니다.

느낀점

대규모 상태 관리의 핵심은 라이브러리가 아니라 atom 설계, 네이밍 컨벤션, 그리고 책임 분리에 있음을 체감했습니다. 처음에는 atom을 잘게 쪼개는 것이 과해 보였지만, 시간이 지날수록 잘게 쪼갠 atom이 리렌더 최적화와 디버깅에 결정적 도움을 준다는 것을 경험했습니다.

기여도: 프론트엔드 100% (상태 아키텍처 설계 및 구현 주도)

GitHub Actions CI/CD 파이프라인 (S3/CloudFront)

문제 정의

초기에는 로컬에서 빌드한 정적 파일을 수동으로 S3에 업로드하는 방식이었습니다. 빌드 환경이 개발자마다 달라 같은 코드에서 다른 결과물이 나오는 문제가 있었고, 배포 시간이 5~10분씩 소요되며 휴먼 에러(잘못된 폴더 업로드, 환경 변수 누락 등)가 잦았습니다. 특히 dev/production 환경 분리가 명확하지 않아 사고가 발생할 뻔한 적도 있었습니다.

기술적 판단 및 해결

GitHub Actions로 PR 머지 트리거 기반의 자동 배포 파이프라인을 구축했습니다. 환경별로 별도 워크플로우(dev/production)를 작성하고, 각각 다른 환경 변수와 S3 버킷을 매핑했습니다. 빌드 → S3 sync → CloudFront invalidation 순서로 구성하되, CloudFront 캐시 무효화를 빌드 결과물 hash 기반으로 처리해 캐시 갱신 누락을 방지했고, 빌드 결과는 워크플로우 상태와 함께 슬랙 알림으로 팀에 공유했습니다.

결과 및 성과

배포가 코드 머지와 동시에 자동으로 진행되어 휴먼 에러가 사라졌고, 환경별로 안정적인 분리가 가능해졌습니다. 배포 시간도 일정하게 관리되어 긴급 수정이 필요한 상황에서도 안전하게 대응할 수 있게 됐습니다.

느낀점

프론트엔드 배포에서 가장 까다로운 부분은 빌드가 아니라 CDN(특히 CloudFront) 캐시 무효화임을 직접 경험했습니다. 단순히 invalidate를 호출하는 것을 넘어, 어떤 파일을 어떤 시점에 무효화할지 전략을 세우는 것이 핵심임을 배웠고, CI/CD는 단순 자동화가 아니라 팀의 안전망이라는 점이 가장 큰 인사이트였습니다.

기여도: CI/CD 100% (워크플로우 설계 및 구축 단독 진행)

Toss Payments 결제 연동

문제 정의

유료 재무 컨설팅 결제를 위해 PG 연동이 필요했는데, 결제는 단순히 "성공/실패" 두 가지로 끝나지 않는 복잡한 상태 머신입니다. 주문 생성, 결제 시도, 결제 승인, 결제 실패, 환불, 부분 환불 등 다양한 시나리오와 그에 따른 사용자 알림·DB 동기화·UI 갱신을 모두 처리해야 했고, 특히 결제 후 사용자가 페이지를 떠났다 돌아왔을 때의 상태 일관성이 어려운 지점이었습니다.

기술적 판단 및 해결

Toss Payments SDK를 통합하고, 주문→장바구니→결제 요청→승인→완료 까지의 전체 플로우를 별도의 결제 도메인으로 캡슐화했습니다. 결제 요청 직전 서버 측 임시 주문을 생성해 결제 ID와 매핑하고, 결제 결과 콜백 URL에서 서버에 검증을 요청해 클라이언트의 상태 변조 가능성을 차단했습니다. 결제 실패·취소 시나리오는 에러 코드별로 사용자에게 다른 안내 메시지를 노출하도록 했습니다.

결과 및 성과

주문·결제·환불 플로우가 안정적으로 동작했고, 결제 실패 시에도 사용자에게 명확한 다음 액션을 안내할 수 있어 결제 포기율이 낮아졌습니다. 결제 상태 검증을 서버 측에서 하도록 분리한 덕분에 보안 위험도 차단할 수 있었습니다.

느낀점

PG 연동은 SDK를 붙이는 작업이 아니라 결제 상태 머신을 정확히 모델링하는 작업임을 절감했습니다. 또한 클라이언트와 서버 사이에서 어디까지 신뢰할지 명확히 정의하는 것(특히 금액 검증)이 보안의 핵심 임을 배웠습니다.

기여도: 프론트엔드 100% (결제 도메인 설계 및 통합 단독 구현)

CreditConnect는 이커머스 사업자를 위한 대출 중계 플랫폼으로, 사용자 그룹에 따라 세 가지 오피스 구조로 설계되었습니다. FO(Front Office)는 쿠팡·네이버스토어 등을 운영하는 판매 사장님들이, BO(Back Office)는 회사 내부 운영 직원이, PO(Partner Office)는 금융사 직원이 사용합니다.

판매몰을 운영하는 셀러는 카드결제·할부 등의 이유로 판매금이 바로 지급되지 않고 정산까지 시간이 걸립니다. 고객이 '구매확정'을 하면 정산예정금이 잡히는데, CreditConnect는 이 정산예정금을 담보로 대출을 신청할 수 있도록 금융기관과 셀러를 연결해 줍니다. 일반적으로 금융기관이 증빙자료가 부족한 사업자에게 판매몰 자료를 일일이 요청하기 어렵다는 문제를 해결하기 위해, CreditConnect는 각 판매몰 어드민에 접속해 데이터를 자동 스크래핑하고 대출 시작부터 종료까지 매일 최신 자료를 금융기관에 제공하며 중간다리 역할을 수행합니다. 이 데이터 중개로 발생하는 이용 요금이 서비스의 핵심 수익 구조입니다.

React 18 SPA 트라이 포탈 아키텍처

문제 정의

대출 중개 플랫폼은 고객(셀러)·운영(내부)·금융 파트너(은행) 등 완전히 다른 권한과 인터페이스를 가진 3개 사용자군을 동시에 지원해야 했습니다. 단순히 권한 분기로 처리하기에는 화면 구조와 도메인 로직이 너무 달랐고, 별도 앱으로 분리하면 공통 로직과 UI 컴포넌트의 재사용성이 떨어지는 딜레마가 있었습니다. 또한 각 포탈을 독립적으로 빌드·배포할 수 있어야 했습니다.

기술적 판단 및 해결

React 18 SPA 기반의 트라이 포탈 구조를 설계하고, 공통 모듈(API 클라이언트, UI 컴포넌트, 유틸리티)을 별도 워크스페이스로 분리했습니다. 각 포탈은 자체 라우트, 권한 정책, 비즈니스 로직을 가지면서도 공통 모듈을 import해 재사용하는 구조를 채택했습니다. 빌드 시 entry point를 분리해 포탈별 독립 배포가 가능하게 했고, 라우트는 140개 이상, JS 파일은 1,350개 이상에 달하는 규모로 성장했지만 명확한 경계 덕분에 유지보수가 가능했습니다.

결과 및 성과

3개 포탈이 공통 코드를 효율적으로 공유하면서도 독립적으로 개발·배포 가능한 구조를 확보했습니다. 특정 포탈의 변경이 다른 포탈에 영향을 주지 않아 사이드 이펙트를 최소화할 수 있었고, 신규 기능 추가 시 공통 모듈에 한 번만 작성하면 모든 포탈에서 활용 가능해 개발 효율이 크게 향상됐습니다.

느낀점

멀티 앱 아키텍처에서 가장 어려운 건 "어디까지 공유하고 어디부터 분리할 것인가"의 경계 설정임을 체감했습니다. 너무 많이 공유하면 결합도가 올라가 변경에 취약해지고, 너무 분리하면 중복이 발생합니다. 이 균형은 코드 단위가 아니라 도메인 경계 기반으로 접근해야 함을 배웠습니다.

기여도: 아키텍처 설계 및 구축 주도 (FE 단독 구현)

86+ 재사용 UI 컴포넌트 & SearchTable 패턴

문제 정의

관리자 도구 성격이 강한 운영 포탈에는 30개 이상의 페이지가 있었고, 대부분 "필터 조건 입력 → 검색 → 결과 테이블 → 행 선택 → 상세/수정" 이라는 동일한 패턴을 반복하고 있었습니다. 페이지마다 비슷한 UI를 매번 새로 짜다 보니 일관성이 떨어지고 개발 시간도 누적되며 중복 코드가 많아졌고, 신규 페이지 추가에 평균 3~5일이 소요됐습니다.

기술적 판단 및 해결

86개 이상의 재사용 UI 컴포넌트(Input, Table, Modal, Dropdown, DatePicker 등)를 디자인 시스템으로 정리하고, 그 위에 config-driven SearchTable 패턴을 도입했습니다. 페이지 개발자는 columns, filters, fetch 함수만 객체로 선언하면 검색·페이지네이션·정렬·행 선택·CSV 내보내기까지 자동으로 동작하는 구조를 만들었습니다. 페이지마다 다른 커스텀 동작은 render prop과 hooks를 통해 확장 가능하게 했습니다.

결과 및 성과

30개 이상 페이지에서 개발 시간이 평균 60% 단축됐고, UI 일관성이 자동으로 유지돼 디자인-개발 간 커뮤니케이션 비용도 줄었습니다. 신규 페이지 추가가 1~2일 단위로 가능해졌고, 기존 페이지 수정도 패턴 변경 한 번으로 일괄 적용 가능해 유지보수성이 크게 향상됐습니다.

느낀점

관리자 도구처럼 반복 패턴이 많은 도메인에서는 config-driven 추상화가 강력한 생산성 도구임을 체감했습니다. 다만 추상화 수준이 너무 높으면 예외 케이스 대응이 어려워지므로, 컴포넌트는 단순하게 두고 확장 포인트를 명확히 노출하는 것이 핵심이라는 점을 배웠습니다.

기여도: 컴포넌트 설계 및 구현 주도

코드 리뷰 프로세스 도입

문제 정의

초기 개발 단계에서는 코드 리뷰 없이 머지하는 문화가 일반적이었고, 그 결과 동일한 결함이 반복적으로 QA에서 발견됐습니다. 한 사람이 작성한 코드를 다른 사람이 이해하지 못해 인수인계 시 시간이 크게 소요됐고, 전체 코드베이스의 일관성도 떨어졌습니다. 유지보수 단계에서 한 부분 수정이 다른 부분을 깨는 사고가 자주 발생해 팀 전체의 신뢰도가 낮은 상태였습니다.

기술적 판단 및 해결

PR 기반 코드 리뷰 프로세스를 도입하고, 팀 내에 리뷰 가이드라인(체크리스트, 컨벤션, 우선순위)을 작성·공유했습니다. 단순히 "리뷰 의무화"가 아니라 어떤 부분을 어떻게 봐야 하는지 명시했고, 리뷰어와 작성자 간 토론이 활발해질 수 있도록 코멘트 컨벤션도 제정했습니다. 매주 회고에서 리뷰 사례를 공유해 학습 문화를 만들었습니다.

결과 및 성과

QA 단계 결함이 약 50% 감소했고, 유지보수 공수는 약 70% 절감됐습니다. 무엇보다 팀 내 코드 수준이 평균적으로 상향 평준화되어 어떤 코드라도 누구나 이해하고 수정할 수 있는 구조가 만들어졌고, 신규 입사자의 온보딩 시간도 단축됐습니다.

느낀점

코드 리뷰는 단순한 품질 검사가 아니라 팀의 지식을 공유하고 문화를 만드는 가장 효과적인 도구임을 절감했습니다. 도구나 프레임워크 도입 보다 "리뷰 문화"라는 무형의 자산이 코드 품질에 미치는 영향이 훨씬 크다는 점이 가장 큰 배움이었습니다.

기여도: 프로세스 설계 및 도입 주도

JWT 자동 갱신 (Request Queue 패턴)

문제 정의

금융 서비스 특성상 JWT 토큰의 짧은 만료 시간이 보안적으로 중요했지만, 사용자가 작업 중 토큰이 만료되면 동시에 여러 API 요청이 401로 실패하면서 화면이 깨지거나 세션이 갑자기 끊기는 문제가 발생했습니다. 단순히 401 발생 시 토큰을 갱신하는 방식으로는 동시 다발 요청에서 토큰 갱신 요청이 중복으로 발생하는 Race Condition이 생겼습니다.

기술적 판단 및 해결

Axios 인터셉터에 Request Queue 패턴을 도입했습니다. 401 응답을 감지하면 첫 요청만 실제 리프레시 토큰 갱신을 시도하고, 그 외 동시 발생한 401 요청은 큐에 보관합니다. 갱신이 성공하면 큐에 쌓인 요청을 새 토큰으로 자동 재시도하고, 실패하면 일괄 reject 처리합니다. 이 모든 과정이 호출 측 코드에서는 완전히 투명하게 동작하도록 추상화했습니다.

결과 및 성과

사용자 입장에서는 토큰 만료를 전혀 인지하지 못한 채 작업을 이어갈 수 있게 됐고, 금융 데이터 접근이 무중단으로 처리됐습니다. 토큰 갱신 요청 중복이 사라져 서버 부하도 감소했고, 401로 인한 화면 깨짐 이슈가 완전히 사라졌습니다.

느낀점

인증 플로우에서 가장 위험한 건 단순한 로직 오류가 아니라 동시성으로 인한 Race Condition임을 직접 체감했습니다. 또한 사용자가 인지하지 못하는 인프라 레벨의 처리가 결국 가장 좋은 UX임을 배웠습니다 — 보이지 않는 곳에서 잘 동작하는 것이 진짜 잘 동작하는 것입니다.

기여도: 설계 및 구현 100%

다계층 권한 관리 시스템

문제 정의

금융 서비스는 화면 단위(접근 가능 페이지), 메뉴 단위(보이는 메뉴), 요소 단위(버튼·필드 표시 여부)로 세밀한 접근 제어가 필요했습니다. 역할이 5개 이상이고 각 역할마다 권한 조합이 달랐으며, 권한 정책이 자주 변경되는 상황에서 코드 곳곳에 권한 체크 if문이 흩어지면 유지보수가 불가능했습니다. 또한 권한이 잘못 노출되면 금융사고로 직결되는 위험이 있었습니다.

기술적 판단 및 해결

권한 관리를 라우트·메뉴·요소의 3계층으로 분리했습니다. 라우트 레벨은 AuthRouteOutlet(로그인 필수 영역), ProtectedRoute(역할별 접근 제어), withoutAuth(비로그인 전용 영역)를 조합한 React Router v6 Outlet 기반 구조로 처리했습니다. 메뉴 레벨은 역할별 접근 가능 화면 리스트를 기반으로 사이드바를 동적으로 생성했고, 요소 레벨은 권한 훅으로 버튼·필드·액션 단위 가시성을 제어했습니다. 모든 권한 정책은 설정 파일 한 곳에 모아 관리해, 신규 권한 추가는 코드 변경 없이 설정만으로 가능하게 했습니다.

결과 및 성과

역할별 세밀한 접근 제어가 가능해졌고, 권한 변경 요청이 와도 설정 파일 수정만으로 대응할 수 있어 변경 비용이 크게 줄었습니다. 권한 누락으로 인한 보안 사고 위험이 사라졌고, 신규 역할 추가도 기존 코드를 수정하지 않고 가능해 안정성이 향상됐습니다.

느낀점

권한 시스템은 초기 설계가 거의 모든 것을 결정한다는 점을 절감했습니다. 후행적으로 권한을 추가하면 코드 곳곳에 if문이 늘어나며 결국 무너지지만, 처음부터 설정 기반 구조로 잘 설계해두면 무한히 확장 가능합니다. 보안 영역일수록 "코드가 아닌 데이터로 표현"하는 접근이 안전하다는 것을 배웠습니다.

기여도: 설계 및 구현 100%

판매몰 스크래핑 연동 시스템 (CS 문의 30% 감소)

문제 정의

대출 적격 심사를 위해서는 쿠팡·네이버스토어 등 마켓플레이스의 셀러 매출 데이터를 자동으로 수집해야 했습니다. 그러나 판매몰 연동 과정은 로그인 단계부터 2차 인증, 보안 문자, 데이터 수집까지 다양한 실패 지점이 존재했고, 사용자는 "왜 연동이 실패했는지" 알 수 없어 CS 문의가 폭증했습니다. 특히 판매몰 측 정책 변경이나 일시적 차단으로 인한 실패도 잦아 상태 추적이 매우 까다로웠습니다.

기술적 판단 및 해결

백엔드 스크래핑 API와 연동해 판매몰 연동 프로세스를 5단계 상태로 모델링했습니다(로그인 시도 → 2차 인증 → 데이터 수집 → 검증 → 완료). 각 단계별로 50여 개의 에러 코드를 정의하고, 에러 코드마다 사용자가 스스로 해결할 수 있는 안내 메시지를 매핑했습니다. 진행 상황은 실시간으로 UI에 표시하고, 실패 시 재시도 플로우와 문의 가이드를 단계별로 노출하도록 설계했습니다.

결과 및 성과

사용자가 어느 단계에서 실패했는지, 다음에 무엇을 해야 하는지 명확하게 파악할 수 있게 되어 CS 문의가 약 30% 감소했습니다. 운영팀은 에러 코드로 문제를 즉시 특정할 수 있게 되어 대응 시간이 단축됐고, 셀러 입장에서도 대출 프로세스가 "블랙박스"에서 "단계별 가시화"로 바뀌어 서비스 신뢰도가 향상됐습니다.

느낀점

외부 시스템 연동에서 가장 중요한 것은 "성공 케이스"가 아니라 "실패 케이스의 명확한 전달"임을 체감했습니다. 동일한 에러도 기술 메시지를 그대로 노출하면 혼란을 주지만, 사용자 관점의 언어로 매핑하면 CS 비용을 크게 줄일 수 있다는 것을 배웠습니다. 에러 설계가 결국 제품 품질의 핵심 축이라는 인식을 얻었습니다.

기여도: 프론트엔드 100% (상태 모델링 및 에러 코드 체계 설계)
